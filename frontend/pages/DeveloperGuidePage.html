Architecture

The IDMS uses Flask in main.py, SQLite accessors in database.py, static Jinja HTML pages in frontend/pages, and unittest-based regression tests at repository root. Data flow is browser form or JSON request to Flask route, database helper validation, SQLite write/read, then HTML or JSON response.

Database Schema

users_data -> votes, meeting_attendance, uploads voting_events -> voting_options -> votes voting_events -> eligibility_audit whatsapp_messages -> analytics queries meetings -> meeting_attendance, meeting_minutes financial_transactions -> financial_report budgets -> budget monitoring notifications -> delivery/history log activity_log -> admin activity summary bug_reports -> bug workflow

Primary Endpoints

  • POST /api/voting/events, POST /api/voting/votes, GET /api/voting/events/<id>/results, GET /api/voting/events/<id>/results.csv
  • POST /api/whatsapp/import, GET /api/whatsapp/analytics
  • GET/POST /meetings, GET/POST /api/meetings, GET /api/meetings/attendance.csv
  • GET/POST /financial, GET /api/financial/report, GET /api/financial/report.csv
  • GET /api/admin/activity-summary, GET/POST /bugs, GET/POST /notifications

Security Notes

Admin-only routes use the existing admin_required decorator. Voting uses a unique event/member constraint to prevent duplicates and stores a bcrypt hash instead of plain ballot values. File import endpoints validate file type and parse only expected text formats.
